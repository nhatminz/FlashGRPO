# FlashGRPO

FlashGRPO is a GRPO training path that replaces the external FastGRPO draft
model with MEDUSA-style heads attached to the policy model.

The default v1 path is:

- Qwen/Llama-style causal LM policy with LoRA GRPO updates.
- MEDUSA-1 head-only online training.
- Frozen backbone for MEDUSA loss.
- Concurrency-aware dense candidate tree.
- Optional uncertainty-guided top-k scheduler inside the concurrency budget.
- Optional Chain-MEDUSA proposal mode for prefix-conditioned deeper candidates.
- `exact_target` acceptance, so appended rollout tokens are sampled from the
  target policy distribution.
- Default KV update via `cache_update_mode: extract_path`; `recompute_accepted`
  is now a debug/fallback mode.

## Difference From FastGRPO

FastGRPO uses a separate draft model, target verification, and online draft
learning. FlashGRPO uses extra heads on top of the policy hidden state:

1. The target LM head samples the root token `y1`.
2. MEDUSA heads propose future candidates `y2, y3, ...`.
3. A tree-attention target forward verifies candidate nodes.
4. The decoder walks a path by sampling from target logits at each parent.
5. A future token is accepted only if the target-sampled token is in the
   MEDUSA candidate children.

The MEDUSA heads never directly decide an output token in the main path.

## Adaptive Tree And Chain-MEDUSA

The main config enables two proposal-side improvements while keeping target
verification unchanged:

- `adaptive_tree_enabled`: computes MEDUSA/chain confidence and shrinks the
  per-depth top-k only when proposal confidence is high. `B_cur` still defines
  the maximum budget through `cpeak_nodes`; confidence only chooses how much of
  that budget is used for the next tree.
- `proposal_mode: chain`: after `chain_enable_after` rollout batches, deeper
  proposal nodes are generated by a lightweight recurrent `ChainDraftCell`.
  Depth 2 is bootstrapped from pretrained MEDUSA head 1, and deeper states are
  conditioned on previously proposed path tokens.

These signals are never used for acceptance. Every proposed node is still
verified by the target model and accepted through the `exact_target` path.

## Run On RTX 3090

First measure a conservative tree budget if desired:

```bash
CUDA_VISIBLE_DEVICES=0 PYTHONPATH=. python flashgrpo/scripts/measure_cpeak.py \
  --model models/Qwen2.5-1.5B-Instruct \
  --dtype fp16 \
  --out logs/flashgrpo/cpeak_measure.json
```

Pretrain MEDUSA heads on ShareGPT:

```bash
CUDA_VISIBLE_DEVICES=0 PYTHONPATH=. python flashgrpo/scripts/pretrain_medusa_heads.py \
  --config flashgrpo/configs/pretrain_medusa_heads_sharegpt_qwen25_1p5b.yaml
```

This writes:

```text
outputs/flashgrpo_medusa_sharegpt_qwen25_1p5b/
  medusa_heads.pt
  medusa_config.json
  medusa_config.yaml
  pretrain_metrics.jsonl
  pretrain_summary.txt
```

Smoke test:

```bash
CUDA_VISIBLE_DEVICES=0 PYTHONPATH=. python flashgrpo/scripts/run_smoke_test.py \
  --config flashgrpo/configs/flashgrpo_qwen25_1p5b_simplelrabel3to5.yaml
```

Main training:

```bash
EXP=flashgrpo_qwen25_1p5b_simplelrabel3to5
MEDUSA=outputs/flashgrpo_medusa_sharegpt_qwen25_1p5b

CUDA_VISIBLE_DEVICES=0 PYTHONPATH=. python flashgrpo/scripts/train_flashgrpo.py \
  --config flashgrpo/configs/flashgrpo_qwen25_1p5b_simplelrabel3to5.yaml \
  --set run_name=${EXP} \
  --set model.model_dir=$MODEL \
  --set flashgrpo.medusa_heads_checkpoint=${MEDUSA} \
  --set logging.log_dir=logs/flashgrpo/${EXP} \
  --set training.saved_model_dir=outputs/${EXP}/target_lora \
  --set training.saved_medusa_dir=outputs/${EXP}/medusa_heads
```

For a safer first run on a single 3090, keep:

- `flashgrpo.num_medusa_heads: 3`
- `flashgrpo.tie_lm_head: true`
- `flashgrpo.max_tree_nodes_per_seq: 16`
- `generation.repeated_generate_nums: 8`
- `training.batch_size: 4`

## Important Configs

- `flashgrpo.acceptance`: default `exact_target`.
- `flashgrpo.typical.enabled`: default `false`.
- `flashgrpo.tree_mode`: `concurrency_aware` or `fixed`.
- `flashgrpo.cpeak_nodes`: total candidate-node budget used by the
  concurrency-aware rule.
- `flashgrpo.adaptive_tree_enabled`: enable confidence-guided top-k shrinkage.
- `flashgrpo.adaptive_confidence_low/high`: confidence range mapping from
  maximum top-k to minimum top-k.
- `flashgrpo.proposal_mode`: `medusa` or `chain`.
- `flashgrpo.chain_enable_after`: rollout warmup before using chain proposals.
- `flashgrpo.chain_loss_weight`: online loss weight for the ChainDraftCell.
- `flashgrpo.medusa_heads_checkpoint`: directory produced by pretraining.
- `flashgrpo.require_pretrained_heads`: default true in the main config.
- `flashgrpo.cache_update_mode`: `extract_path` by default; `recompute_accepted`
  only for debug.
- `flashgrpo.allow_recompute_fallback`: if false, extraction failures raise.
- `flashgrpo.online_medusa`: train heads online after rollouts.
- `flashgrpo.medusa_training_mode`: default `head_only`.
- `flashgrpo.tie_lm_head`: default true to fit Qwen2.5-1.5B on 24GB. Untied
  per-head vocab projections are supported by the module but are much heavier.

## Metrics

Logs are written to:

```text
logs/flashgrpo/<run_name>/
  metrics.jsonl
  metrics.csv
  config_resolved.yaml
  summary.txt
```

Key fields:

- `generation_time`
- `tokens_per_sec_generation`
- `average_accept_length`
- `accepted_tokens_per_medusa_step`
- `medusa_acceptance_rate`
- `tree_nodes_per_seq`
- `B_cur_avg`
- `medusa_loss`
- `head_update_time`
- `head_update_tokens_per_sec`
- `reward_variance`
- `cache_update_mode`
- `kv_extraction_success_count`
- `kv_extraction_fallback_count`
- `kv_extraction_time`
- `recompute_fallback_time`
- `oom_count`
- `gpu.*`

Baseline speedup fields are not faked. If baseline measurement is not enabled,
`baseline_source` is `not_available` and speedup is null.

## Limitations In v1

- Tree attention uses the eager attention backend for arbitrary masks.
- KV extraction supports the Qwen/HF DynamicCache path used by this repo.
  Unsupported cache formats either fallback with metrics or raise when
  `allow_recompute_fallback=false`.
- Sparse tree is a config skeleton and maps to dense in v1.
- Typical acceptance is intentionally disabled in the main path.
- Tied LM head is the recommended 3090 setting; untied output heads can be too
  memory-heavy for online GRPO.
